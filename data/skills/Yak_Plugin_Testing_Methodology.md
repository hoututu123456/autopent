# Yak / Yakit 插件测试方法论（以组件漏洞检测为例）

## 目标
用“可复核的证据链”验证检测插件/规则的正确性：既能命中真实漏洞，也能控制误报与边界条件。

## 基本策略
1. 明确插件类型
   - 资产识别类：版本/指纹/组件存在性
   - 配置缺陷类：未授权、弱配置、错误暴露
   - 漏洞检测类：基于可验证行为的探测（避免破坏性利用）
2. 建立对照组
   - 漏洞版本与修复版本
   - 功能相近但不同产品/不同端点（用于检验误报）
3. 证据闭环
   - 插件输出至少包含：目标、探测点、请求关键字段、响应关键片段（可脱敏）
   - 可选：截图/日志路径
4. 边界条件归纳
   - 不同部署形态（反向代理、WAF、不同 URL 前缀）
   - 不同鉴权/权限模型（匿名/用户/管理员）
   - 频率限制与错误处理
5. 沉淀复用
   - 检测边界写入 Knowledge
   - 复现与排障写入 Playbook

## 组件漏洞检测清单示例：WebLogic 插件页能提供什么价值
WebLogic 插件集合展示了“同一组件多个检测点”的组织方式，适合借鉴为：
- 组件检测矩阵：按 CVE/功能点分类
- 插件 ID 与测试说明：用于搭建复测脚本与证据标准

建议在平台内沉淀两类内容：
- Knowledge：每类 CVE 的“受影响范围/典型入口/常见误报点/修复建议”
- Playbook：在靶场或授权环境中复测的“最小验证步骤 + 证据采集”
