# Yakit Web Fuzzer：FuzzTag 与批量发包工作流

## 目标
把“重放 + 爆破/模糊测试”的常见流程变成可复用、可读、可回放的操作步骤，降低配置成本并提升测试覆盖度。

## 核心概念：FuzzTag
FuzzTag 是一类轻量语法结构，形态类似：
`{{tagname(tagvalue)}}`

特点：
- 类似函数调用，但作为“模板占位符”存在于请求中
- 可嵌套（例如将一个 Tag 的输出交给另一个 Tag 处理）
- 适合在请求的多个位置插入，从而实现批量发包与参数组合

## 典型用法（非破坏性测试优先）
### 1) 单参数枚举
用于探测参数边界、类型、长度与简单枚举场景。

### 2) 多参数组合
当多个参数都存在输入面时，可在不同位置插入不同 Tag，形成组合测试矩阵。

### 3) 分组（实现“并行对齐”的字典迭代）
当需要“多个字段按相同索引配对”时，可通过分组标识让多个位置共享同一轮次的值（从而避免笛卡尔积爆炸）。

## 常用测试场景映射（建议）
- SQLi/NoSQLi：从“最小探测”开始（注入特征与报错差异），再升级到更细粒度枚举
- XSS：优先输出编码/反射点定位，再转向上下文敏感 payload（避免在生产环境执行危险脚本）
- 命令注入：优先验证分隔符与回显通道（回显/时延/侧信道），再做严格证据闭环
- 认证与会话：优先做速率限制、锁定策略、MFA/验证码逻辑验证（避免未经授权的口令爆破）

## 抓包联动建议
在 Web Fuzzer 前建议先通过代理/抓包模块获得“真实请求”：
- 确认关键 headers、cookies、token、CSRF 方案
- 确认请求体编码（JSON/form/multipart）与字段名
- 先重放 1 次保证基线可用，再引入 FuzzTag

## 输出与沉淀（与 AutoPentestAI 对齐）
建议把一次成功的测试沉淀为 Playbook：
- 测试入口与先决条件（鉴权/角色/业务状态）
- 请求模板（含变量与说明，不写可直接攻击的武器化载荷）
- 判定准则（响应差异、日志证据、截图路径）
- 修复建议（校验、编码、WAF/网关、限速）
